# Product Spec

## Overview

I am a manager of a group of Product Managers. I’d like a system that helps me manage the many projects and products within my sphere of influence. It will have a dashboard that shows me important stats about each project and product. I imagine each project as a card with a title and various visualizations and data within it. Those cards are within other cards, which represent the primary product they're serving, but may have lines which represent connections to products. The product card may have stats on it as well, at the bottom. Those products are within still larger cards which represent my areas of responsibility (such as home, work, etc.). Projects may not always have a product. They may just be within areas of responsibility by themselves, examples of this may be "Admin Tasks" or "Intake Requests"

## Project Card

Each project card will have the title of the project centered on the top of the card in larger font. Just under the title will be the objective of the project. The far left of the card will have four squares which show the count of tasks by what quadrant they are in (1, 2, 3, 4) The number will be dark green if one or more of them is a focus item for the week. the top left square is Quadrant 1, top right square is Quadrant 2, bottom left square is Quadrant 3, bottom right square is Quadrant 4 or tasks without a quadrant assignment. Archived or deleted tasks are not included in the count. The far right of the card will show the Owner name and delivery due date. The date will be black if estimated, green if committed and on track, orange if committed and at risk, red if committed and missed (impacted), blue if done. The center of the card will show Risks and open questions. In the upper right is an expand icon which will show all of the same information as on the card, plus more details, and, at the bottom, the same task table as described below. The user may also edit anything from this view. Additional details will include the reporter(s) and stakeholder(s) on the risks and open questions, Notes (with a date-added and a stakeholder name list on each, if given), Project stakeholders (organized by department), problem statement, Activity, Impact, and Outcome metrics (forecast and measured), attachments, and links. The user may also add a new field in this view and select whether they want it to be added to all projects, just this project, and whether any field should be displayed on the card (this action is not available for the list of fields that I described as having a designated place on the card. New fields that should be displayed on the card are listed below the permanent fields. In this same view, the user should be given the option to archive or delete the project. Projects that are archived are not shown on the main page. Tasks within an archived project are also archived.

## Task Table Popup

When a person clicks on a task number, they'll see a table popup that shows the status of the task (Not Started, In Progress, Delegated, Pending Feedback, On Hold, Blocked, Done, Delegated-Done), quadrant of the task (with additional details such as H for highly important, Q for quick task, or HQ for both, if applicable. For example one might be in quadrant 1, another 1H, or another 2HQ, etc.) , the type of tasks (Communication, Open Questions, Scrum Needs, Employee Needs, Other), a description of the task, notes on the task, stakeholders, and attachments. For each task, a user can mark it as a focus item and/or one of today’s top 3. At the bottom of the table will be a blue row that says "New Task" with a plus sign on the left. A task can be created with any field but description blank. Tasks can also be archived or deleted. Archived tasks can be viewed from the expanded view of the project.

## Product Card

Product cards will have a title and a description on the top left. Stakeholders will be listed in the center. In the upper right is an expand icon which will show all of the same information as on the card, plus more details if the user has added more fields. It will also give the user the same capability to add new fields as on the project cards.

## Area of Responsibility Card

Area of responsibility cards will have a title and a description on the top left. In the upper right is an expand icon which will show all of the same information as on the card, plus more details if the user has added more fields. It will also give the user the same capability to add new fields as on the project cards.

## Create New & Filter

At the top of the dashboard is a "Create New" button. When clicked, the user can select to create a Task, Project, Product, or Area of Responsibility. When one of those is selected, a popup will show that allows the user to create any one of those and fill in any details they have. Next to the button is a filter button, where a person can filter out any Area of Responsibility, Product, or Project

## Right Sidebar — Focus Feed & Status Chart

On the far right of the page will be a list of today’s top three on the top and all of this week’s focus item tasks, organized by project and type. The user can change the status of the task from this list or from within the drop-down in the card mentioned above. Tasks can also be archived or deleted from here. Underneath this list is a horizontal bar chart which shows task status counts by project. Each status has a different color, all colors are light, Done or Delegated – Done is green, Delegated is blue, In Progress is yellow, Blocked is dark grey, Pending Feedback is orange, On Hold is red, tasks with no status are represented in a white bar with a thin black border.

## Left Sidebar — Stakeholders

On the far left of the page is a list of stakeholders by department with an avatar showing either their image (if provided) or their initials (if not provided). Next to each stakeholder will be check boxes (tasks, notes, projects, products). Inside the check box will be the number of such things that the stakeholder is listed on. For example, if three tasks have the person as one of the stakeholders, the number 3 will show in that bock. Clicking any of those will highlight the relevant object if it has the stakeholder’s name attached to it. For example, the quadrant of the task, or the row of the task if it’s being shown in any of the three places it can be (upon clicking the quadrant, the expand button, or the list on the far right of the page). An “Add Stakeholder” button show above the list. Clicking the button will allow the user to add a stakeholder’s name, department, and image. Name and department must be added, image is optional.

## Chat Assistant

On the lower right of the page is a chat icon. When clicked, a text box will float above everything on the lower fifth of the page (with reasonable margins around). The chat icon will show in the upper left of the text box, a submit button will show on the lower right. A pus icon will show on the lower left. A collapse icon will show on the upper right. When the user clicks the chat icon, they can choose among different icons or add their own (the chat icon would change both in the text box and on the main page when the text box is not expanded). When they click the plus button they can add attachments. Typing adds text to the text box. Clicking the collapse button removes the text box (but text is retained for next time they click the chat icon again) The submit button submits the text to an LLM which will interpret what they said and take action. Actions that can be taken are adding tasks to a project, projects to a product (or area of responsibility), products to an area of responsibility, adding a new area of responsibility, adding a new stakeholder, adding any level of detail to or modifying any of these objects, adding note (with the datetime and stakeholder captured, if given) designating a task as one of today’s top three, or really any other kind of action that the user would be able to take on the page. The user may also ask the LLM about any information contained in any of the objects, even if archived. The user may also chat with the LLM about any other topic. Responses from the LLM are shown in a text bubble above the text field. An understated nav bar to the right of the response can be used to scroll when the response is long, or scroll to previous parts of the message history. Message history is retrieved via lazy loading, so that the memory on the page is not overused. When the LLM is responding, the chat icon does some cool looking animation. Chat history is saved and can be looked up by clicking a left caret “<” under the chat icon. Clicking that caret pops us a card to the left of the text boxes that floats above the rest of the page, with previous saves sessions.

## Archive View

In the lower left of the page, a “view archive” button will provide a date range picker and reveal all projects that were or are active in that date range, and all of their data, even those archived. So as not to hog memory, when viewing the archive tasks are retrieved only when one of the task tables is being displayed, although the four quadrant counts are shown and maintained accurately. Archived objects (tasks, projects, etc.) can be edited just like active objects. When viewing an archived project, the button that would otherwise enable archiving is changed to a “Restore Project” button and will un-archive the project.
